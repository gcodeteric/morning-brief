# Dashboard Data Architecture

## What this layer does

The internal Streamlit dashboard should not re-discover operational state by scraping unrelated files on every page.

`dashboard_data.py` centralizes that work by:

- loading the latest run summary
- resolving the latest brief path and text
- loading manual overrides safely
- loading card outputs safely
- loading a lightweight latest-run snapshot when available
- returning a normalized dashboard context with stable keys

## Snapshot

The dashboard snapshot is a lightweight JSON file written to:

- `data/dashboard_latest_snapshot.json`

This file already existed in the current dashboard integration, so the path was kept for backward compatibility with:

- the existing dashboard code
- current `.gitignore`
- the latest successful run already present on disk

## Why the snapshot exists

Some useful runtime state is not persisted anywhere else in a clean structured form, especially:

- final planner selections
- Instagram morning/afternoon digest variants
- digest packs generated by the agent pipeline
- top curated stories with direct links
- resolved card paths
- override summary attached to the final plan

Without the snapshot, the dashboard can still load:

- `data/run_summary.json`
- the latest brief markdown
- `data/manual_overrides.json`
- cards from the output folder

But it cannot reliably reconstruct the full structured state of the latest run.

## Snapshot contents

The snapshot is intentionally lightweight. It currently stores:

- `snapshot_kind`
- `schema_version`
- `timestamp`
- `run_status`
- `curated_stories`
- `curated_categories`
- `total_before_dedup`
- `total_after_dedup`
- `plan`
- `agent_outputs`
- `instagram_morning_pack`
- `instagram_afternoon_pack`
- `card_paths`
- `brief_path`
- `override_summary`
- `manual_overrides_applied`
- `run_summary`

The markdown brief body is **not** duplicated inside the snapshot. The dashboard loads that separately from the brief path.

## Write behavior

The snapshot is written from `main.py` only on normal successful runs.

Dry runs do **not** refresh the snapshot. This is intentional:

- a dry run should not overwrite the last real operational state
- the dashboard keeps pointing at the last successful real run

Snapshot persistence is non-critical:

- if writing fails, the main pipeline continues
- the brief still gets generated normally

Writes are atomic when practical:

- the JSON is written to a temporary file
- then replaced into place

## Fallback behavior

`dashboard_data.py` uses this order:

1. load snapshot if present and valid
2. fall back to direct operational artifacts where possible

Fallback mode still provides safe access to:

- brief path/text
- run summary
- overrides
- cards folder
- runtime/config status

If structured digest data is missing, the dashboard should show empty/default sections instead of crashing.

## Main public helpers

Key helpers exposed by `dashboard_data.py`:

- `load_latest_run_summary()`
- `load_latest_brief_path()`
- `load_latest_brief_text()`
- `load_manual_overrides()`
- `load_cards()`
- `load_latest_snapshot()`
- `load_runtime_status()`
- `load_instagram_digest_data()`
- `load_other_channel_data()`
- `build_dashboard_context()`

Backward-compatible aliases are preserved for the current dashboard code:

- `load_dashboard_snapshot()`
- `load_dashboard_context()`

## Normalization

The module also normalizes common dashboard objects so the UI can rely on stable fields:

- `normalize_story()`
- `normalize_digest()`
- `normalize_card_paths()`
- `is_valid_snapshot()`

This keeps direct-link behavior reliable by ensuring stories expose, where possible:

- `title`
- `source`
- `category`
- `score`
- `summary`
- `link`
